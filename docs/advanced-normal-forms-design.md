# BCNF / 第4正規形 / 第5正規形 教材設計メモ

## 目的

既存の `非正規形 → 1NF → 2NF → 3NF` の学習体験を壊さずに、その先の `BCNF → 4NF → 5NF` を追加する。

この教材では定義の暗記よりも、次の順で理解を進める。

1. 1つの表に複数の事実を押し込む
2. 編集・追加・削除で矛盾や面倒さが起きる
3. なぜ起きるかをヒントで見る
4. 分解後の表で「どこが楽になったか」を確かめる

## 全体方針

- 既存の 3NF までは「注文 / 顧客 / 商品 / 営業」の流れで統一されている。
- BCNF 以降は概念が急に難しくなるため、1つの大きな受注業務を無理に引き延ばすより、各正規形に合う小さな業務場面へ切り替える方が教材として分かりやすい。
- ただし世界観は `販売・受注まわりの業務` に寄せ、まったく別教科に見えないようにする。
- 既存 UI の強みである `試してみよう`、`異常検出`、`構造ヒント`、`変更箇所の赤字表示` をそのまま活かす。

## BCNF

### ねらい

- 3NF を満たしていても、`候補キーでない列が他の列を決めてしまう` と更新異常が残ることを理解する。
- `決定項は候補キーであるべき` という BCNF の感覚をつかむ。

### 教材で体験させたい問題

- 同じ担当営業が複数の顧客行に現れるため、訪問曜日の変更を何度も直す必要がある。
- 担当営業ごとに訪問曜日が決まるのに、その事実が行ごとに重複している。
- ある行を消すと、`担当営業に割り当てられた訪問曜日` という事実まで消える。

### サンプル業務場面

`顧客訪問割当`

業務ルール:

- 1人の営業担当には固定の訪問曜日が割り当てられている。
- 1人の顧客は複数の営業担当と関わることがある。
- したがって `担当営業ID -> 訪問曜日` が成り立つ。

### サンプルテーブル案

テーブル名: `顧客訪問割当`

主な列:

- `顧客ID`
- `訪問曜日`
- `担当営業ID`

候補キーの想定:

- `(顧客ID, 訪問曜日)`
- `(顧客ID, 担当営業ID)`

初期データ例:

| 顧客ID | 訪問曜日 | 担当営業ID |
| --- | --- | --- |
| C-01 | 火曜 | S-01 |
| C-02 | 火曜 | S-01 |
| C-01 | 木曜 | S-02 |
| C-03 | 木曜 | S-02 |

この表では `担当営業ID -> 訪問曜日` があるため、`担当営業ID` は決定項だがスーパーキーではない。
一方で `訪問曜日` は候補キー `(顧客ID, 訪問曜日)` に含まれる主属性なので、3NF と BCNF の違いを見せやすい。

### 分解後テーブル案

1. `顧客_担当営業`
   - 列: `顧客ID`, `担当営業ID`
2. `担当営業`
   - 列: `担当営業ID`, `訪問曜日`

初期データ例:

`顧客_担当営業`

| 顧客ID | 担当営業ID |
| --- | --- |
| C-01 | S-01 |
| C-02 | S-01 |
| C-01 | S-02 |
| C-03 | S-02 |

`担当営業`

| 担当営業ID | 訪問曜日 |
| --- | --- |
| S-01 | 火曜 |
| S-02 | 木曜 |

### 体験させる操作例

- `S-01` の訪問曜日を 1 行だけ `水曜` に変える
- `S-02` を担当する最後の行を削除して、担当営業の曜日情報そのものが消えるか確認する
- 新しい担当営業 `S-03` を追加しようとして、顧客割当がないと登録しづらいことを見る

### UI 上の見せ方

- ミッション文:
  `この表では、担当営業ID が訪問曜日を決めています。1つだけ書き換えてズレが起きるか試してください。`
- 構造ヒント:
  `顧客ID + 訪問曜日 -> 担当営業ID` だけでなく、`顧客ID + 担当営業ID -> 訪問曜日` と `担当営業ID -> 訪問曜日` も図で見せる。
- 図の見せ方:
  `候補キーが複数ある` ことと `候補キーでない決定項` を同時に示す。
- 異常検出:
  同じ `担当営業ID` に複数の `訪問曜日` が出たらハイライトする。
- 成功メッセージ:
  `担当営業ごとの訪問曜日を 1 か所で管理できるようになりました。`

### 実装メモ

- `hint.kind = 'bcnf'` を追加し、`X -> Y` のうち `X が主キーではない` ことを赤系アクセントで示す。
- `試してみよう` は既存と同じ 3 件程度で十分。

## 第4正規形

### ねらい

- 1つの主キーに対して `独立した多値の事実` が 2 系統あると、組合せの行が大量に必要になることを理解する。
- `A さんが知っている商品` と `A さんが使う連絡手段` のように、互いに無関係な多値属性は別表に分けるべきだと学ぶ。

### 教材で体験させたい問題

- 仕入先を 1 社追加するだけで、販売チャネルの数だけ行を増やさないといけない。
- 1 行削除しただけなのに、`この商品はこのチャネルでも売る` という別の事実まで失いやすい。
- 表の行が `独立事実の直積` になっていて、重複が本質的に避けられない。

### サンプル業務場面

`商品ごとの仕入先と販売チャネル`

業務ルール:

- 1つの商品は複数の仕入先から仕入れられる。
- 同じ商品は複数の販売チャネルで売られる。
- `仕入先` と `販売チャネル` は独立しており、互いに決め合わない。

### サンプルテーブル案

テーブル名: `商品_仕入先_販売チャネル`

主な列:

- `商品ID`
- `仕入先`
- `販売チャネル`

初期データ例:

| 商品ID | 仕入先 | 販売チャネル |
| --- | --- | --- |
| P-01 | 北果物商事 | 店舗販売 |
| P-01 | 北果物商事 | EC販売 |
| P-01 | 南青果センター | 店舗販売 |
| P-01 | 南青果センター | EC販売 |
| P-02 | 山手乳業 | 店舗販売 |
| P-02 | 山手乳業 | 定期配送 |

この表は `商品ID ->-> 仕入先` と `商品ID ->-> 販売チャネル` を同時に持つ。

### 分解後テーブル案

1. `商品_仕入先`
   - 列: `商品ID`, `仕入先`
2. `商品_販売チャネル`
   - 列: `商品ID`, `販売チャネル`

初期データ例:

`商品_仕入先`

| 商品ID | 仕入先 |
| --- | --- |
| P-01 | 北果物商事 |
| P-01 | 南青果センター |
| P-02 | 山手乳業 |

`商品_販売チャネル`

| 商品ID | 販売チャネル |
| --- | --- |
| P-01 | 店舗販売 |
| P-01 | EC販売 |
| P-02 | 店舗販売 |
| P-02 | 定期配送 |

### 体験させる操作例

- `P-01` に新しい仕入先 `中央青果` を追加しようとして、チャネル数ぶんの行追加が必要になることを確認する
- `P-01 × 北果物商事 × EC販売` の 1 行だけ削除し、仕入先の事実とチャネルの事実が中途半端に欠ける状態を見る
- `P-01` の新しい販売チャネル `催事販売` を追加しようとして、全仕入先との組合せ入力が必要になることを見る

### UI 上の見せ方

- ミッション文:
  `この表では、仕入先の情報と販売チャネルの情報が独立なのに、すべての組合せで行が増えています。追加や削除がどれだけ面倒か試してください。`
- 構造ヒント:
  `商品ID ->-> 仕入先` と `商品ID ->-> 販売チャネル` を左右に分けて表示する。
- 図の見せ方:
  1つの中心ノード `商品ID` から 2 方向へ多値矢印を伸ばす新ヒント種別を追加する。
- 異常検出:
  同一商品について、仕入先集合と販売チャネル集合の直積が崩れているか、または直積そのものが過剰重複を生んでいることを説明する。
- 成功メッセージ:
  `仕入先の追加と販売チャネルの追加を別々に管理できるようになりました。`

### 実装メモ

- `hint.kind = 'multivalued'` を追加する。
- 異常検出は厳密な 4NF 判定より、次を優先する。
  - 同一商品で仕入先が複数、チャネルも複数あることを検出
  - そのために組合せ行が増えていることを説明
  - 直積の一部だけ欠けた場合に `独立事実を1表で持つと管理が難しい` と返す

## 第5正規形

### ねらい

- 3つの主体の関係が、実は `3つの二者関係` の組合せとして表せる場合があることを理解する。
- そのような関係を 1 つの 3 項関係表で持つと、追加・削除時に複数の事実が絡んでしまうと学ぶ。

### 教材で体験させたい問題

- 1つの三者組を消しただけで、`商品を扱える販売店`, `販売店が契約する配送業者`, `その配送業者がその商品を運べる` のいずれの事実を消したかったのか分からない。
- 新しい三者組を追加するとき、本当は既にある二者関係の再表現なのに 3 列同時に埋める必要がある。
- 3項関係の表だけを見ていると、どの二者関係が本質なのか見えにくい。

### サンプル業務場面

`商品・販売店・配送業者の取扱可否`

業務ルール:

- ある販売店がその商品を扱える
- ある販売店がその配送業者と契約している
- ある配送業者がその商品を配送できる
- この 3 条件がそろうと三者組が成立する

この教材では、上のルールが成り立つようにデータを固定し、`3つの二者関係に分けられる三者関係` を扱う。

### サンプルテーブル案

テーブル名: `商品_販売店_配送業者`

主な列:

- `商品ID`
- `販売店ID`
- `配送業者ID`

初期データ例:

| 商品ID | 販売店ID | 配送業者ID |
| --- | --- | --- |
| P-01 | ST-01 | D-01 |
| P-01 | ST-01 | D-02 |
| P-01 | ST-02 | D-01 |
| P-02 | ST-01 | D-01 |

この例から次の二者関係を読み取れるようにする:

- `商品_販売店`
- `販売店_配送業者`
- `商品_配送業者`

### 分解後テーブル案

1. `商品_販売店`
   - 列: `商品ID`, `販売店ID`
2. `販売店_配送業者`
   - 列: `販売店ID`, `配送業者ID`
3. `商品_配送業者`
   - 列: `商品ID`, `配送業者ID`

初期データ例:

`商品_販売店`

| 商品ID | 販売店ID |
| --- | --- |
| P-01 | ST-01 |
| P-01 | ST-02 |
| P-02 | ST-01 |

`販売店_配送業者`

| 販売店ID | 配送業者ID |
| --- | --- |
| ST-01 | D-01 |
| ST-01 | D-02 |
| ST-02 | D-01 |

`商品_配送業者`

| 商品ID | 配送業者ID |
| --- | --- |
| P-01 | D-01 |
| P-01 | D-02 |
| P-02 | D-01 |

### 体験させる操作例

- `P-01 × ST-01 × D-02` を削除し、`ST-01 は D-02 と契約している` 事実まで同時に失うか考える
- `P-02 × ST-01 × D-02` を追加して、実は 3 つの二者関係がそろえば作れる組合せであることを確認する
- 3項表だけでは何の事実を管理しているのか読み取りにくいことを比較する

### UI 上の見せ方

- ミッション文:
  `この表は3者の組合せをそのまま並べています。本当に管理したい事実が、二者関係に分けられないか確認してください。`
- 構造ヒント:
  三角形の 3 辺に `商品-販売店`, `販売店-配送業者`, `商品-配送業者` を置く。
- 図の見せ方:
  新しい `join dependency` の図が必要。
- 異常検出:
  3項表の行から導ける二者関係が重複しており、1行の削除がどの事実を消したい操作なのか曖昧だと説明する。
- 成功メッセージ:
  `三者関係を、意味のある二者関係として分けて管理できるようになりました。`

### 実装メモ

- `hint.kind = 'join'` を追加する。
- 5NF の説明は難しいため、モーダル内に短い補助文を入れる。
  - `この教材では、三者関係が3つの二者関係の結合で表せる場合だけを扱います。`
- 異常検出は `厳密な 5NF 証明` ではなく、`三者表に二者関係の事実が混ざっていて、削除の意味が曖昧` であることを返す。

## UI 共通改善案

### 1. ステージ導入文の補強

BCNF 以降は概念が難しいため、各ステージの説明に次の 2 点を毎回入れる。

- `この段階で増えた新しい見方`
- `前の正規形では防げなかった問題`

### 2. ヒント図の拡張

追加したい `hint.kind`:

- `bcnf`
- `multivalued`
- `join`

### 3. 成功状態の表現

3NF までは `重大な異常なし` で十分だったが、BCNF 以降は次のように少し具体化するとよい。

- BCNF: `決定する事実の置き場所が1か所になりました`
- 4NF: `独立した多値事実を別々に追加できます`
- 5NF: `三者組ではなく、意味のある二者関係を管理できます`

### 4. 試してみようボタンの役割

BCNF 以降は、手で操作しても問題の意味が見えにくい。
そのため `試してみよう` はほぼ必須に近い導線として扱う。

推奨:

- 各ステージ 3 件
- `1件だけ書き換える`
- `1行だけ消す`
- `新しい事実を追加する`

## 補足

- BCNF / 4NF / 5NF は厳密定義まで一気に出すと初学者には重い。
- 先に `この表だと、何を1回で直せないのか` を体験させ、その後でヒントモーダルに用語を出す流れを維持する。
- 実装では `完全な理論判定` より `教材として納得できる異常の見え方` を優先する。
